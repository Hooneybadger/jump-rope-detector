# 모아뛰기 라벨-영상 정합성 분석 보고서

## 1. 목적

본 문서는 `/label`의 Kinovea 라벨과 `/video`의 원본 영상을 기반으로,

- 라벨의 의미
- 타임스탬프 정합성
- 1회 카운트의 정의
- MediaPipe 기반 관측 가능 신호

를 분석하여 **실시간(e2e) 모아뛰기 카운터 전략을 수립**하는 것을 목표로 한다.

분석 범위는 구현이 아니라 검증과 전략 수립이다. 특히 다음을 최우선으로 두었다.

- Kinovea 타임스탬프와 실제 영상 시간축의 정합성 확인
- 라벨 의미를 실제 데이터로 확인
- 1카운트의 운영적 정의 도출
- MediaPipe 기반 실시간 전략만 고려

분석 환경 확인 결과 사용 가능한 자원은 `16 CPU`, `31 GiB RAM`, `가용 메모리 약 27 GiB`, `디스크 여유 약 911 GiB`였다. 메타데이터 추출, 샘플 프레임 검증, MediaPipe 라벨 프레임 추론은 이 자원을 활용해 병렬/대량 확인했다.

## 3. Kinovea 라벨 구조 분석

### 3.1 파일 구조

- 포맷: Kinovea 2025.1.1이 저장한 XML (`<KinoveaVideoAnalysis>`)
- 주요 필드:
  - `OriginalFilename`
  - `FullPath`
  - `ImageSize`
  - `AverageTimeStampsPerFrame`
  - `CaptureFramerate`
  - `UserFramerate`
  - `SelectionEnd`
  - `Keyframes/Keyframe`
- 타임스탬프 필드: `Keyframe/Timestamp`
- 좌표 필드: `Keyframe/Drawings/Pencil/PointList/Point`
- 트랙 구조:
  - 별도 left/right track이 아니라, 보통 **한 Keyframe 안에 2개의 Pencil**이 들어 있다.
  - 각 Pencil은 거의 동일한 위치의 점을 3~4개 반복 저장하고 있어, 실제 의미는 하나의 발 포인트로 보는 것이 타당하다.
  - 정상 이벤트는 `한 타임스탬프 + 두 발 포인트` 구조다.

---

### 3.2 라벨 의미 해석

각 라벨이 의미하는 것은 무엇인가?

- 왼발:
  - 정상 이벤트에서는 좌측/우측 발 부근에 각각 1개의 점이 찍혀 있다.
  - 좌우 발은 별도 시간차 라벨이 아니라 **같은 시점의 양발 이벤트**로 저장된다.
- 오른발:
  - 대부분의 프레임에서 두 점의 y 차이가 매우 작다.
  - 예: 파일별 발 간 y 차이 중앙값은 대체로 `0.5~6 px` 수준이다.

이 라벨은 다음 중 무엇을 의미하는가?

- 착지 시점: 일부 샘플에서 매우 가깝다.
- 점프 시작: 일부 샘플에서 매우 가깝다.
- 공중 상태: 아니다.
- 카운트 인정 시점: 가장 가까운 해석이다.
- 기타: **양발 접지 구간의 바닥점 또는 상승 전환점에 해당하는 "카운트 이벤트 윈도우"**

실데이터 기반 해석은 다음과 같다.

- `01`, `03`, `05`, `09`, `10`, `11` 대표 샘플을 실제 프레임으로 확인했을 때 라벨 포인트는 모두 발등/발끝/앞꿈치 근처에 위치했다.
- 라벨 프레임은 공중 정점이 아니라 **양발이 지면과 접촉해 있거나 막 떨어지기 직전인 구간**에 있었다.
- 샘플 윈도우의 MediaPipe 궤적을 보면, 라벨 시점은 대개 평균 힙 y 또는 평균 발목 y의 **국소 최대값(가장 아래 위치)** 또는 그로부터 `±1~2프레임` 범위에 있다.

중요한 점:

- `Pencil` 순서 자체는 의미론적 left/right가 아니다.
- 대부분 x 순서는 안정적이지만 `05.kva`에서는 1회 순서 뒤바뀜이 있었다.
- 따라서 라벨 XML의 Pencil 순번으로 좌우를 하드코딩하면 안 된다.

### 3.3 타임스탬프 해석

- 단위: **파일군마다 다르다.**
- 정밀도:
  - `01/02/04/08`: ms 기반
  - `03/05/06/07/09/10/11`: 15360 tick 기반
- 기준:
  - 절대 시간: 예
  - 프레임 기반: tick 그룹은 사실상 프레임 정렬
  - UI 보정 시간: 근거 없음

구체적으로는 두 가지 모드가 확인됐다.

1. `ms-like` 그룹: `01, 02, 04, 08`

- `AverageTimeStampsPerFrame=33`
- `SelectionEnd / duration ≈ 1000`
- `Timestamp / 1000 = 초`
- 예: `01.kva` 첫 라벨 `Timestamp=3203`, `Name=3.20`
- 이 그룹은 영상이 30fps인데 라벨 헤더의 framerate는 `29.97002997003`인 경우가 있어, 프레임 매핑 시 `±0.5 frame` 정도의 양자화 오차가 생긴다.

2. `tick-like` 그룹: `03, 05, 06, 07, 09, 10, 11`

- `AverageTimeStampsPerFrame=512`
- `512 tick/frame`, `30 fps`, 즉 `15360 tick/sec`
- `SelectionEnd / duration ≈ 15360`
- 모든 키프레임 `Timestamp`가 `512`의 배수였다.
- 즉 이 그룹은 `frame = Timestamp / 512`로 **정확히 프레임 인덱스에 떨어진다.**
- 예: `03.kva` 첫 라벨 `Timestamp=20992`, `20992/512=41 frame`, `Name=1.37`

결론적으로 라벨 타임스탬프는 하나의 공통 규칙이 아니라 **파일군별 시간축 변환이 필요한 절대시간/프레임 혼합 표기**다.

---

### 3.4 의심되는 문제

- 같은 데이터셋 안에서 타임스탬프 단위가 `ms`와 `15360 tick`로 섞여 있다.
- `10.kva`는 `OriginalFilename`과 `FullPath`가 실제 대응 영상과 다르다.
- 구조 이상치가 존재한다.
  - `01.kva` 1개 이벤트: `Drawings` 자체가 없음
  - `02.kva` 2개 이벤트: 한 이벤트에 발 포인트 1개만 있음
  - `09.kva` 1개 이벤트: 같은 시점에 포인트 4개(중복 기입 추정)
- `02.kva`의 `7854ms`, `7867ms`는 불과 `13ms` 차이로, 사실상 하나의 점프 이벤트가 두 개로 쪼개졌을 가능성이 높다.
- `ms-like` 그룹은 정확한 프레임 정렬이 아니라 시간 정렬이므로, 단일 프레임 정답으로 쓰기에는 본질적 한계가 있다.

## 4. 영상 타이밍 분석

### 4.1 메타데이터

- FPS: 모든 영상이 `ffprobe` 기준 `30/1`
- 길이: `5.767s ~ 212.000s`
- CFR/VFR 여부:
  - `r_frame_rate=30/1`
  - `avg_frame_rate=30/1`
  - `nb_frames`가 duration과 30fps에 일관되게 대응
  - 대표 샘플 `01.mp4`, `10.mp4`의 프레임 `pkt_duration_time`은 연속적으로 `0.033333`으로 확인됨
  - 따라서 **적어도 본 분석 범위에서는 CFR로 보는 것이 타당**
- 프레임 드롭 여부:
  - 메타데이터상 명백한 드롭 징후는 없음
  - 다만 저수준 패킷 전체 전수 검사는 하지 않았으므로, "없음"이 아니라 "발견하지 못함"으로 표현하는 것이 안전함

---

### 4.2 타이밍 리스크

- FPS와 실제 재생이 불일치하는가?
  - 영상 자체는 30fps다.
  - 그러나 `ms-like` 라벨 헤더는 29.97fps 계열이라, 헤더 숫자를 그대로 프레임 변환에 쓰면 드리프트가 생긴다.
- 프레임 간격이 일정하지 않은가?
  - 샘플 확인 범위에서는 일정했다.
- Kinovea와 시간축이 다를 가능성?
  - 있다.
  - 실제로 라벨은 `ms-like`와 `tick-like` 두 축이 섞여 있으므로 **파일별 변환 규칙 적용이 필수**다.

핵심 리스크는 VFR보다도 **라벨 시간축 해석 오류**다.

## 5. 라벨-영상 정합성 분석 (핵심)

### 5.1 정합 방식

라벨 타임스탬프를 실제 영상 프레임/시간과 다음 규칙으로 매칭했다.

1. 파일 매핑

- 기본 매핑은 `label/NN.kva ↔ video/NN.mp4`
- 해상도 일치 여부로 1차 검증
- `10.kva`는 헤더 불일치가 있었지만 실제 영상 크기/길이/좌표 위치로 `10.mp4`와 정합 확인

2. 시간축 변환

- `ms-like` 그룹:
  - `time_sec = Timestamp / 1000`
  - `frame_idx = round(time_sec * 30)`
- `tick-like` 그룹:
  - `time_sec = Timestamp / 15360`
  - `frame_idx = Timestamp / 512`

3. 샘플 검증

- 대표 샘플 프레임을 `라벨 시점 ±4~5 프레임`으로 추출
- 라벨 포인트를 영상 위에 중첩해 실제 발 위치와 맞는지 시각 확인
- 같은 윈도우에 대해 MediaPipe Pose를 돌려 평균 힙/발목 y 궤적이 어디서 바닥점을 갖는지 비교

4. 추가 수치 검증

- `tick-like` 그룹은 모든 라벨 타임스탬프가 `512` 배수여서 프레임 매핑 오차가 `0`
- `ms-like` 그룹은 라벨 시점의 프레임 반올림 오차 중앙값이 최대 `0.265 frame`, 최대 `0.5 frame`

### 5.3 주요 발견

- 일정 오프셋 존재 여부:
  - 샘플 검증 범위에서는 **고정된 전역 오프셋은 발견하지 못함**
- 시간 드리프트 존재 여부:
  - 파일별 올바른 시간축 변환을 적용하면 뚜렷한 드리프트는 보이지 않음
- 좌우 발 라벨 동기 여부:
  - 정상 이벤트는 한 타임스탬프에 두 발이 동시에 기록됨
  - 즉 구조적으로는 **동기형 양발 이벤트**
  - 다만 `02.kva`에는 한 발만 찍힌 예외 2건이 있음
- 특정 패턴 존재 여부:
  - 라벨은 공중 apex가 아니라 **양발 접지 윈도우**에 몰려 있다
  - 샘플 이벤트 72개(6개 영상, 각 12개 샘플)에서
    - 평균 힙 y의 국소 최대가 라벨 프레임 `±1 frame` 안에 들어온 경우: `64/72 (88.9%)`
    - 평균 발목 y의 국소 최대가 라벨 프레임 `±1 frame` 안에 들어온 경우: `52/72 (72.2%)`
  - 영상별 편차는 존재한다.
    - `01`은 라벨보다 `+1~+3 frame` 뒤에 최대가 오는 경우가 잦다.
    - `05`는 라벨과 최대가 같거나 `-1 frame` 앞에 오는 경우가 많다.
    - `09`는 고속 리듬이라 발 곡선이 평평해 일부 샘플에서 최대 위치가 불안정하다.

즉, **라벨은 한 프레임 정밀의 단일 점보다 "접지/반등 이벤트 윈도우"를 가리킨다**고 보는 것이 맞다.

---

### 5.4 결론

라벨 타임스탬프는:

- **신뢰 가능(단, 파일별 시간축 해석 필요)**

이유:

- 영상은 30fps CFR로 보이며, 적절한 변환을 적용하면 라벨 포인트가 실제 발 위치와 맞는다.
- `tick-like` 그룹은 프레임 매핑이 정확하다.
- `ms-like` 그룹은 `±0.5 frame` 정도의 불확실성이 있으나, 카운팅 전략 정의에는 충분하다.
- 다만 구조 이상치(`01`, `02`, `09`)와 헤더 이상치(`10`)는 전처리 예외 규칙으로 처리해야 한다.

## 6. "1 카운트"의 정의

### 6.1 후보 정의

- 착지 기준:
  - 장점: 직관적
  - 단점: 실제 라벨은 순수 착지 1프레임에만 고정되어 있지 않음
- 점프 기준:
  - 장점: 올라가는 순간은 실시간 검출이 쉬움
  - 단점: 라벨은 이륙 단일 프레임보다 접지 윈도우 전체에 더 가까움
- 공중 기준:
  - 장점: 주기 분리가 쉬워 보일 수 있음
  - 단점: 실제 라벨과 가장 멂
- 양발 기준:
  - 장점: 데이터 구조와 일치
  - 단점: 단순 양발 근접만으로는 무릎 굽힘/제자리 바운스를 배제하기 어려움
- 주기 전환 기준:
  - 장점: 라벨 편차를 흡수 가능
  - 단점: 정의를 명확히 해야 함

---

### 6.2 최종 정의

"1 카운트"는 다음 시점으로 정의한다:

→ **양발이 동시에 지면 접지 상태에 있고, 하강이 끝난 뒤 다시 상승으로 전환되는 1회의 반등(contact-to-rebound) 이벤트**

운영상으로는 다음과 같이 해석하는 것이 가장 안전하다.

- 라벨 정답을 단일 프레임으로 고정하지 않고
- **접지 윈도우 내에서의 반등 시작점**을 1카운트로 본다

이 정의는 실시간 시스템에서 `1~2프레임` 지연만으로도 인과적으로 구현 가능하다.

---

### 6.3 근거

- 라벨 패턴:
  - 정상 이벤트는 항상 양발 동시 기록이다.
- 영상 관찰:
  - 실제 샘플 프레임에서 라벨은 공중이 아니라 접지 구간에 있다.
- 일관성 여부:
  - 힙 y 국소 최대가 라벨 `±1 frame` 내에 들어오는 비율이 높다.
  - 발 y는 영상별 리듬과 해상도에 따라 약간 흔들리지만, 전체적으로 접지 구간과 일치한다.
- 주기 길이:
  - 라벨 간 중앙 간격은 대체로 `0.233s ~ 0.500s`
  - 이는 실제 모아뛰기 1회 주기 범위로 해석 가능하다.
  - `09`, `10`은 각각 약 `3.57`, `4.06 count/s`로 빠른 리듬 사례다.

즉, **1개 Keyframe = 1회 모아뛰기 이벤트**로 보는 해석이 데이터와 가장 잘 맞는다.

## 8. 구현된 실시간 카운팅 엔진 기술서

이 절은 분석 결과를 실제 구현으로 옮긴 현재 엔진의 동작 원리를 기술한다. 구현은 `MediaPipe Pose -> signal 추출 -> start gate -> unified counter engine` 순서로 동작한다.

핵심 원칙은 다음과 같다.

- 전체 영상 후처리 없이 프레임을 순차 처리한다.
- centered window나 미래 프레임 peak picking을 사용하지 않는다.
- 1카운트는 `접지 후 반등 전환`으로 정의한다.
- 카운트 타이밍과 오탐 억제는 같은 unified counter engine 안에서 함께 결정된다.

### 8.1 입력 랜드마크와 신호 정의

현재 엔진이 실제로 사용하는 핵심 랜드마크는 아래와 같다.

| 구분 | 랜드마크 | 사용 방식 |
| ---- | -------- | --------- |
| 중심축 | `LEFT_HIP`, `RIGHT_HIP` | 평균하여 `mean_hip_y` 계산 |
| 발 접지축 | `LEFT_ANKLE`, `LEFT_HEEL`, `LEFT_FOOT_INDEX` | 좌측 `foot_y` 계산 |
| 발 접지축 | `RIGHT_ANKLE`, `RIGHT_HEEL`, `RIGHT_FOOT_INDEX` | 우측 `foot_y` 계산 |
| 스케일 정규화 | `LEFT_HIP-LEFT_ANKLE`, `RIGHT_HIP-RIGHT_ANKLE` | 평균 수직 길이로 `leg_length` 계산 |

신호 계산 방식은 다음과 같다.

- `mean_hip_y = (left_hip_y + right_hip_y) / 2`
- `left_foot_y`, `right_foot_y`:
  - 발목, 힐, 발끝 중 visibility가 `0.35` 이상인 점만 우선 평균
  - 보이는 점이 없으면 세 점 전체 평균으로 fallback
- `mean_foot_y = (left_foot_y + right_foot_y) / 2`
- `leg_length`:
  - `(|left_hip_y - left_ankle_y| + |right_hip_y - right_ankle_y|) / 2`
  - 모든 임계치는 이 값으로 정규화한다.
- `symmetry_y = |left_foot_y - right_foot_y|`

관측성 기준:

- 힙 visibility는 좌우 모두 `>= 0.50`일 때만 `signal.detected=True`
- 발은 세 후보점의 평균을 쓰므로, 발 단일 랜드마크 하나가 잠깐 흔들려도 완전히 끊기지 않도록 설계했다.

### 8.2 1카운트는 어떻게 판별하는가

한 번의 점프는 아래 순서를 만족할 때 1카운트로 인정된다.

1. `mean_foot_y`가 현재 바닥 밴드 근처에 있어야 한다.
2. 좌우 발 높이 차가 작아야 한다.
3. `mean_hip_y`가 먼저 하강해야 한다.
4. 그 다음 프레임들에서 `mean_hip_y` velocity가 상승으로 전환되어야 한다.
5. 직전 카운트와 충분한 간격이 있어야 한다.

이 정의를 코드 수준으로 쓰면 다음과 같다.

#### 1. causal smoothing

- `mean_hip_y_raw`, `mean_foot_y_raw`에 EMA를 적용한다.
- 기본 EMA:
  - `hip alpha = 0.45`
  - `foot alpha = 0.55`
- 매우 빠른 cadence에서는 fast EMA로 전환한다.
  - `fast hip alpha = 0.25`
  - `fast foot alpha = 0.35`

#### 2. 바닥 밴드 추정

절대 바닥 좌표를 고정하지 않고, 최근 발 높이로 `floor_y`를 온라인 추정한다.

- 최초 프레임: `floor_y = mean_foot_y`
- 이후:
  - `floor_y = max(mean_foot_y, floor_y - floor_decay_ratio * leg_length)`
  - 기본 `floor_decay_ratio = 0.004`

이렇게 하면 사용자의 신장, 카메라 높이, 약간의 프레이밍 변화에 적응할 수 있다.

#### 3. 접지 게이트

현재 프레임이 "양발 접지 상태"로 볼 수 있는지 다음 조건으로 판단한다.

- `mean_foot_y >= floor_y - 0.08 * leg_length`
- `symmetry_y <= 0.12 * leg_length`

즉, 발이 바닥 밴드 근처에 있고 좌우 발이 너무 비대칭이면 카운트 후보로 보지 않는다.

#### 4. 반등 판정

힙 y velocity 부호로 하강과 상승을 구분한다.

- `descending`:
  - `hip_vel >= 0.006 * leg_length`
- `ascending`:
  - `hip_vel <= -0.004 * leg_length`

매우 빠른 cadence에서는 더 민감한 fast threshold로 전환한다.

- `fast descending = 0.002 * leg_length`
- `fast ascending = -0.001 * leg_length`

#### 5. 상태기계

상태는 `READY -> CONTACT -> REBOUND_LOCK` 세 단계다.

| 상태 | 의미 | 전이 조건 |
| ---- | ---- | --------- |
| `READY` | 아직 접지 반등 이벤트가 확정되지 않음 | 접지 게이트가 참이면 `CONTACT` |
| `CONTACT` | 양발 접지 상태에서 반등을 기다림 | 먼저 `descending`을 봐야 하고, 그 다음 `ascending`이면 카운트 |
| `REBOUND_LOCK` | 방금 한 번 센 직후 중복 방지 구간 | 충분히 회복 후 다음 cycle에서만 해제 |

실제 카운트가 올라가는 조건은 아래와 같다.

- 현재 상태가 `CONTACT`
- 이전에 `descending`을 이미 봄
- 현재 프레임이 `ascending`
- refractory 조건 충족

즉, **양발 접지 상태에서 몸 중심이 아래로 내려갔다가 다시 위로 튀어오르는 첫 전환점**이 1카운트다.

#### 6. 왜 이 시점이 1카운트인가

이 시점은 라벨 의미와 가장 잘 맞는다.

- 순수 공중 apex가 아니다.
- 순수 착지 1프레임도 아니다.
- 접지 구간 안에서의 `반등 시작점`에 가깝다.

따라서 카운트는 접지 이후 `1~2 frame` 안에서 인과적으로 결정된다.

### 8.3 realtime start gate

카운팅은 영상이 시작되자마자 바로 켜지지 않는다. 사용자가 화면 안에 안정적으로 들어왔다고 판단된 뒤에 시작한다.

현재 규칙은 다음과 같다.

- landmark visibility threshold: `0.30`
- ready 조건:
  - 33개 전체 landmark 중 `97% 이상`이 보이는 상태
- ready hold:
  - 위 상태가 `1.0초` 연속 유지되어야 함
- countdown:
  - 그 뒤 `3.0초`
- dropout tolerance:
  - 준비/카운트다운 중 landmark가 잠깐 흔들려도 `0.35초` 이내면 유지

이 단계는 jump count 자체와는 별개로, realtime UX를 위한 시작 안정화 단계다.

### 8.4 unified motion guard

상태기계만으로는 "손만 흔들기", "발만 까딱하기", "한 점프 두 번 세기"를 완전히 막기 어렵다. 현재 구현은 상태기계 뒤에 별도 엔진을 두지 않고, 같은 `RealtimeCounterEngine.step()` 안에서 motion guard를 함께 적용한다.

guard는 아래 윈도우 특징을 사용한다.

- `motion_window_frames = 18`
- `recent_window_frames = 5`
- `hip_range_ratio = (max(hip) - min(hip)) / median(leg_length)`
- `foot_range_ratio = (max(foot) - min(foot)) / median(leg_length)`
- `recent_hip_range_ratio = recent 5 frame 기준 hip range / median(leg_length)`

기본 통과 조건:

- `hip_range_ratio >= 0.045`
- `foot_range_ratio >= 0.035`
- `recent_hip_range_ratio >= 0.032`
- `foot_to_hip_ratio <= 2.5`
- `min_gap >= 9 frame`

역할은 다음과 같다.

- `hip_range_ratio`: 발만 움직이고 몸 중심이 안 움직이는 경우 차단
- `foot_range_ratio`: 손만 움직이고 발 접지 신호가 약한 경우 차단
- `recent_hip_range_ratio`: 실제 반등이 아니라 작은 드리프트일 때 차단
- `foot_to_hip_ratio`: 발 signal만 과도하게 큰 꼬리 구간이나 발장난 차단
- `foot_dominant_low_hip`: 발 range는 큰데 힙 반등은 약한 구간 차단
- `min_gap`: 같은 점프 안에서 두 번 세는 경우 차단

중요한 점:

- 이 guard는 realtime 전용 별도 엔진이 아니다.
- dataset evaluator와 realtime runner가 같은 `RealtimeCounterEngine`을 사용한다.
- 따라서 현재 count emission logic은 평가와 실사용이 동일하다.

### 8.5 빠른 cadence 적응 로직

실사용에서 가장 큰 문제는 빠른 리듬에서 고정 기준이 너무 보수적이라는 점이었다. `02_realtime`처럼 실제 jump 간격이 `13~15 frame` 수준인 경우, 고정 gap과 과도한 recent-hip 기준이 undercount를 만들었다.

이를 해결하기 위해 unified engine은 아래 조건에서 동적으로 완화된다.

#### cadence lock 조건

- 최근 `motion-valid candidate`의 hip/foot range 중앙값이 충분히 커야 한다.
  - `candidate hip median >= 0.06`
  - `candidate foot median >= 0.05`
- 최근 candidate interval 기록이 최소 1개 이상 있어야 한다.

여기서 `motion-valid candidate`는 다음을 만족하는 상태기계 이벤트다.

- `hip_range_ratio >= 0.045`
- `foot_range_ratio >= 0.035`

즉, 아무 후보나 interval 학습에 쓰지 않고, 손-only/발-only 성격이 약한 후보만 cadence 학습에 사용한다.

#### adaptive min gap

cadence lock이 걸리면:

- `raw_interval_median = median(recent candidate intervals)`
- `adaptive_min_gap = clamp(round(0.8 * raw_interval_median), 5, 9)`

동시에 recent hip 조건도 약간 완화한다.

- `adaptive_recent_hip_floor = 0.032`

이 완화는 무제한으로 풀지 않고, 빠른 cadence에서 필요한 최소 수준만 낮추는 방식이다.

### 8.6 트러블슈팅 기록

실제 realtime 시연에서 관찰된 문제와 해결 방식은 아래와 같다.

| 문제 | 원인 | 해결 방식 | 결과 |
| ---- | ---- | --------- | ---- |
| 손만 움직여도 count | 힙 velocity sign-change만으로 event가 발생할 수 있음 | `foot_range_ratio`와 recent hip 검사 추가 | 손-only 오탐 감소 |
| 발만 움직여도 count | 발 signal만 크고 힙 반등은 약함 | `hip_range_ratio`, `foot_to_hip_ratio`, `foot_dominant_low_hip` 추가 | 발-only 오탐 차단 |
| 한 점프에서 2번 count | 같은 rebound 안에서 지역적 velocity 전환이 두 번 발생 | `REBOUND_LOCK` + `min_gap` 적용 | double-count 감소 |
| 빠른 cadence에서 undercount | 고정 gap이 실제 리듬보다 큼 | candidate cadence 기반 adaptive gap 도입 | `02_realtime`에서 빠른 연속 점프 복구 |
| counting 시작이 너무 늦음 | all landmarks 완전일치 + single-frame dropout 즉시 리셋 | ready ratio 방식과 `0.35s` dropout tolerance 도입 | start gate 지연 대폭 감소 |

#### 사례 1. 빠른 cadence undercount

사용자 시연 `02_realtime`에서 실제 20회를 수행했는데 9회만 카운트된 사례가 있었다.

분석 결과:

- 상태기계 raw 후보는 충분히 나왔지만
- unified engine의 고정 gap 기준이 실제 cadence보다 커서
- 실제 `13~15 frame` cadence의 점프를 절반 가까이 버리고 있었다.

해결:

- `motion-valid candidate interval` 기준 adaptive gap 도입
- `adaptive_recent_hip_floor`를 둬 빠른 cadence에서만 recent-hip 완화

결과:

- 저장 시연 영상 `02_realtime.mp4` 재실행 기준 `final_count=20`

#### 사례 2. counting start 지연

저장 시연 영상 재실행에서 `COUNTING`이 `15.00s`에 시작되는 문제가 있었다. 이는 목표 UX와 맞지 않았다.

원인:

- 준비 조건이 사실상 `33개 landmark 전부 visible`과 같았고
- countdown 중 한 프레임만 흔들려도 다시 `SEARCHING`으로 리셋됐다.

해결:

- ready 조건을 `visible ratio >= 0.97`로 변경
- 준비/카운트다운 중 `0.35초` 이내 flicker는 허용

결과:

- 같은 저장 영상 기준 `COUNTING @ 15.00s -> 5.70s`
- 사용자의 원래 live 로그 `COUNTING @ 6.76s`와도 크게 벗어나지 않는 수준으로 복구

### 8.7 현재 구현에서의 한계

- unified motion guard는 설명 가능성과 실시간성을 우선한 heuristic이다.
- irregular cadence, 한두 번의 큰 헛동작, 카메라 흔들림이 섞이면 추가 튜닝이 필요할 수 있다.
- 저장된 overlay mp4 재생과 실제 live camera 입력은 Pose 품질이 완전히 같지 않다.
- `01_realtime` 같은 음성/비정상 동작 샘플은 더 모아 regression set으로 관리하는 것이 바람직하다.

## 9. 리스크 및 불확실성

- 라벨은 "접지 이벤트 윈도우"에는 일관되지만, 모든 영상에서 동일한 단일 프레임 규약으로 찍힌 것은 아니다.
- 현재 dataset 평가는 `first_label - 15 frame`에서 시작하고, `last_label + 3 frame`까지 허용하는 쪽이 라벨 의미와 더 잘 맞았다.
- `ms-like` 그룹은 본질적으로 `±0.5 frame` 수준의 불확실성이 있다.
- `02.kva`, `09.kva`의 구조 이상치는 전처리 없이 학습/평가에 넣으면 왜곡을 만든다.
- `10.kva`는 헤더 메타데이터 오류가 있어, 향후 자동 매칭 로직은 헤더가 아니라 실제 파일명/해상도/길이를 우선해야 한다.
- 현재 realtime 엔진은 설명 가능한 heuristic 기반으로 동작하므로, 다른 카메라 구도와 조명에 대한 일반화는 별도 검증이 필요하다.
- start gate는 개선되었지만, 전신 landmark visibility에 여전히 일부 의존한다. 실제 제품화 단계에서는 `핵심 관절 subset ready gate`가 더 적절할 수 있다.
- MediaPipe 라벨 프레임 관측성은 매우 좋았지만, 이는 양성 샘플 중심 분석이다. 비점프, 부분 가림, 카메라 흔들림, 발 잘림이 포함된 부정 사례 평가는 아직 충분하지 않다.
